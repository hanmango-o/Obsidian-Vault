# 학습 문서 스킬

## 개요

이 스킬은 학습 노트의 작성, 질의응답 기반 보강, 문서 간 링크 연결을 위한 3가지 모드를 제공합니다.

## 모드

| 모드 | 설명 | 트리거 |
|------|------|--------|
| **writing** | 새 문서 작성 및 QnA 기반 문서 보강 | 문서 작성 요청 시 |
| **tutor** | 문서 기반 질의응답, QnA 섹션에 기록 | "튜터 모드", "질문있어" 등 |
| **link** | 기존 문서들과 Obsidian 링크 연결 | "링크 걸어줘", "문서 연결해줘" 등 |

---

# Writing 모드

## 사용 시점

- 새로운 학습 노트 작성 시
- 기술 개념에 대한 설명 문서 작성 시
- `Study/` 디렉토리 내 문서 생성 시
- QnA 섹션 내용을 본문에 반영하여 문서 보강 시

## 문서 구조

문서는 반드시 아래 순서와 구조를 따릅니다.
별도의 제목 없이 바로 본론으로 진행합니다.

```
ex. 파일 제목 : Activity 생명주기
# Activity 생명주기 -> X
바로 What Is 섹션 bullet으로 시작 -> O
```

---

### 1. YAML Frontmatter

```yaml
---
modified: YYYY-MM-DD
topic: 주제
---
```

주제는 해당 md 파일이 위치한 폴더 명입니다.
```
ex. Android > Activity 생명주기.md
topic: Android
```

---

### 2. What Is 섹션

문서 최상단에 Bullet으로 나열합니다. 별도의 섹션 표시 없이 바로 시작합니다.
섹션 마지막에 구분선(`---`)을 넣어 본문과 구분합니다.

포함 내용:
- Bullet으로 최대 10개 내외 작성
- 이 문서에서 무엇을 배우는지
- 어디까지 다루는지 (범위)
- 이 문서를 읽으면 무엇을 알 수 있는지

**예시:**
```markdown
- ViewModel의 개념과 역할
- ViewModel의 생명주기
- ViewModel과 Activity/Fragment의 관계
- Configuration Change에서의 동작 방식
- ViewModel 생성 및 사용 방법

---
```

---

### 3. 본문 섹션

#### Heading 규칙
- `##` - 대주제
- `###` - 중주제
- `####` - 소주제
- `#####` 이상은 사용하지 않음

#### 작성 규칙
- **~습니다 체** 사용
- 장황하지 않게 간결하게 작성
- 독자가 읽기 쉽도록 구성

#### 코드 작성
- 코드 블록: 언어 명시 필수
  ```kotlin
  val viewModel = ViewModelProvider(this)[MyViewModel::class.java]
  ```
- 인라인 코드: 문장 내 짧은 코드는 `` ` `` 사용

#### 다이어그램
- 그림이 필요한 경우 **mermaid** 활용

#### 테이블
- 비교/요약이 필요한 경우 테이블 활용

#### 링크
- URL 그대로 사용 금지
- 반드시 anchor text 사용: `[설명](URL)`

---

### 4. 정리 섹션

- `## 정리` 형식으로 시작
- **~습니다 체 사용 금지** - 단일 명사/명사구로 작성
- What Is 섹션과 대응되어야 함
- 장황하지 않게 핵심만 요약

**좋은 예시:**
```markdown
## 정리

- ViewModel의 역할: UI 상태 관리 및 비즈니스 로직 처리
- 생명주기: Configuration Change에서도 데이터 유지
- 권장 패턴: 단방향 데이터 흐름
```

**나쁜 예시:**
```markdown
## 정리

- ViewModel은 UI 상태를 관리합니다.
```

---

### 5. QnA 섹션

- `## QnA` 형식으로 시작
- Tutor 모드에서 작성된 질의응답 내용
- `Q1.` / `A1.` 형식으로 번호 부여

```markdown
## QnA

Q1. ViewModel은 언제 사용해야 하나요?
A1. UI 관련 데이터를 Configuration Change에서도 유지해야 할 때 사용합니다.
```

---

### 6. QnA 기반 문서 보강

사용자가 요청 시 QnA 섹션의 내용을 분석하여 본문에 반영합니다.

- 자주 묻는 질문 → 본문에 설명 추가
- 보충 설명이 필요한 부분 → 해당 섹션 확장
- 새로운 개념 → 새 섹션 추가

# Tutor 모드

## 사용 시점

- "튜터 모드", "질문있어", "이거 궁금해" 등의 트리거
- 현재 문서 내용에 대한 질문이 들어올 때

## 동작 방식

1. 사용자가 현재 문서에 대해 질문
2. 문서 내용을 기반으로 답변 제공
3. 질문과 답변을 QnA 섹션에 추가

## QnA 작성 규칙

```markdown
## QnA

Q1. [사용자 질문]
A1. [답변 - 문서 내용 기반으로 명확하게]

Q2. [사용자 질문]
A2. [답변]
```

- 번호는 순차적으로 부여
- 답변은 문서 내용을 기반으로 작성
- 문서에 없는 내용이면 그 사실을 명시하고 일반 지식으로 답변

## 예시

**사용자:** onCreate와 onStart의 차이가 뭐야?

**동작:**
1. 답변 제공
2. QnA 섹션에 추가:
```markdown
Q3. onCreate와 onStart의 차이가 뭐야?
A3. onCreate는 Activity가 생성될 때 최초 1회 호출되며 초기화 작업을 수행합니다. onStart는 Activity가 화면에 보이기 시작할 때마다 호출됩니다.
```

# Link 모드

## 사용 시점

- "링크 걸어줘", "문서 연결해줘", "위키링크 추가해줘" 등의 트리거
- 문서 작성과 튜터링이 완료된 후
- 사용자가 더 이상 수정이 필요없다고 판단했을 때

## 동작 방식

1. `Study/` 디렉토리 내 다른 문서들 탐색
2. 현재 문서에서 다른 문서와 연관된 키워드 찾기
3. Obsidian wikilink 형식으로 링크 추가

## 링크 규칙

- obsidian-skills 참고
- Obsidian wikilink 형식: `[[문서명]]`
- 별칭 사용 가능: `[[문서명|표시텍스트]]`
- 본문 내 관련 키워드에 자연스럽게 삽입

## 예시

**변경 전:**
```markdown
Activity의 생명주기를 이해하면 ViewModel을 효과적으로 사용할 수 있습니다.
```

**변경 후:**
```markdown
[[Activity 생명주기|Activity의 생명주기]]를 이해하면 [[ViewModel]]을 효과적으로 사용할 수 있습니다.
```

## 링크 대상

- 같은 topic 폴더 내 문서 우선
- 관련 개념 문서 (예: Activity ↔ Fragment ↔ ViewModel)
- 이미 존재하는 문서만 링크 (없는 문서 링크 금지)

# 행동 제약

1. **구조 준수**: 반드시 정해진 섹션 순서를 따름
2. **구분선 필수**: 각 주요 섹션 사이에 `---` 삽입
3. **URL 금지**: Raw URL 사용 금지, anchor text 필수
4. **Heading 제한**: `####`까지만 사용
5. **정리 형식**: 명사구로 작성, ~습니다 체 금지
6. **코드 블록**: 언어 명시 필수
7. **QnA 형식**: Q1./A1. 번호 형식 사용
8. **Link 모드**: 존재하는 문서만 링크
